# Symphony Readiness and Bounded Canary Implementation Plan

Version: v1.5

Date: 2026-07-22

Status: closed at Slice 1 (`NO-GO`)

## Goal

Reopen the previously blocked Symphony path only far enough to determine,
with current host and tracker evidence, whether the maintained engine can own
one bounded continuation loop safely. Establish a reusable host capability;
after generic host and scratch-tracker gates pass, use a no-product-change
`bio_plot_platform` task as the first repository-owned canary payload.

## Non-goals

- Do not change the calibration skill, harness guidance, behavior cases, or
  historical evaluation fixtures.
- Do not build a custom scheduler, pull-request babysitter, or Symphony fork.
- Do not require every repository to adopt Symphony, `WORKFLOW.md`, staging,
  or identical delivery configuration.
- Do not encode `bio_plot_platform` labels, validation commands, Kubernetes
  access, deployment topology, or rollback procedure in calibration.
- Do not enable a recurring service, broad personal credentials, auto-merge,
  or autonomous staging access during readiness validation.
- Do not attempt E03/E04 before the scratch and no-product-change gates pass.

## Ownership

- The host control plane owns the pinned Symphony runtime, isolated service
  identity, per-task workspaces, bounded process lifetime, logs, dashboard,
  and cleanup evidence.
- A disposable private GitHub repository owns the adapter live-E2E fixture.
- `bio_plot_platform` owns any later `WORKFLOW.md`, task identity, repository
  validation, pull-request, staging, and rollback contract.
- Calibration owns only this cross-project adoption decision and its closeout.

## Slice 1: Ephemeral Host Readiness

Use a temporary root and the exact current upstream commit. First verify the
official remote ref and checked-out commit, unchanged upstream lockfile, and
checksummed toolchain bootstrap. Perform dependency resolution and audit inside
a disposable constrained build identity that cannot read user credentials or
unrelated repository state. Immediately after resolution, and before compiling
or running dependency code, audit the complete lockfile. Only after that gate
passes, verify Erlang and Elixir versions, focused GitHub-adapter tests,
upstream `make all`, and a Codex App Server protocol handshake. Do not write a
service, persistent secret, or repository configuration in this slice.

The selected source is Symphony commit
`1f3219bb1ea5f69a1305dc594e79b0db57c113c5`. A different commit requires a
fresh identity and test readback rather than silently advancing the pin.

Run the pinned Hex version's `mix hex.audit` immediately after dependency
resolution and before any task that loads, compiles, or starts dependency or
application code. Preserve its advisory identifiers and severity output and
read back the linked Hex or GitHub security records for every finding.
Separately check the Erlang, Elixir, and toolchain versions. If the installed
Hex command cannot provide lockfile advisory and severity coverage, use a
pinned lockfile scanner that can; an unavailable or incomplete scanner does
not pass the gate. Record every advisory and its affected and fixed versions.
Any unmitigated critical or high advisory in a dependency that the readiness
checks or deployed runtime would execute is a `NO-GO` before the affected build
or test task, credentials, tracker writes, a dashboard listener, or persistent
installation. Medium and lower findings require an applicability decision and
explicit acceptance rather than being hidden by a successful `make all`.

Stop with `NO-GO` if the build, tests, App Server handshake, dependency safety,
source provenance, lockfile integrity, or host isolation boundary cannot be
established.

## Slice 2: GitHub Scratch Live E2E

Use a dedicated private scratch repository and a fine-grained GitHub identity
limited to repository metadata and Issue read/write. Do not reuse the user's
broad GitHub CLI OAuth token as the Symphony worker credential.

Run both the upstream GitHub live-E2E test and the subsequent foreground,
runtime-limited orchestrator smoke inside the same tested dedicated OS or
container identity. Neither worker may share the unrestricted `fqzhang` host
namespace or be able to read the user's SSH, GitHub CLI, Kubernetes, Docker,
or unrelated Codex state; an alternate `CODEX_HOME` under the same unrestricted
Unix UID is not sufficient.

Verify that an unlabeled issue is untouched; the required label enables one
claim; label removal stops dispatch; re-adding it permits logical resume; and
closure reaches terminal cleanup. For in-process continuation, record the
same issue and workspace plus the same Codex thread ID and a new turn ID. For
process-restart recovery, record the same issue and deterministic workspace;
a new thread ID is logical recovery and must not be called same-session
continuation. No branch, pull request, ruleset, Actions, or repository-content
write may occur.

Any unauthorized repository access, duplicate worker or external write,
secret exposure, non-loopback listener, unbounded retry, failure to stop, or
workspace escape is a critical `NO-GO`.

## Slice 3: Bounded Host Canary

Only after Slices 1 and 2 pass, install immutable tested program files under
the `fqzhang` user for a reversible canary. Keep program, configuration,
workspace, and state roots separate. The Symphony process, every workspace
hook, and every Codex child must run inside the tested dedicated OS or
container identity; the host `fqzhang` process may launch and stop that boundary
but must not execute workflow logic or hold the worker's runtime credentials.
Do not rely on the host user's file permissions as the credential boundary.
Run one disabled-by-default service with:

- `Restart=no`;
- `RuntimeMaxSec=10min`;
- `TimeoutStopSec=30s`;
- `KillMode=control-group`;
- `UMask=0077`;
- a single-instance lock; and
- a dashboard bound only to `127.0.0.1`.

Mount or expose only an isolated Codex runtime directory and the explicitly
selected tracker credential. Do not mount the Docker socket or expose the
user's general GitHub, SSH, Kubernetes, or other repository credentials to the
worker.

Acceptance requires unit verification, a proven runtime timeout, loopback-only
listening, successful state API readback, clean stop, and no remaining canary
process or listener. The unit remains disabled after the slice.

## Slice 4: `bio_plot_platform` No-product Canary

Only after the generic gates pass, add a minimal repository-owned workflow
through a separate `bio_plot_platform` pull request. Before running it, read
back that `WORKFLOW.md` explicitly defines task selection, isolated workspace
ownership, repository-owned validation, status reporting, and terminal
cleanup. Run one issue whose task is limited to cloning, reading local
authority, executing the repository's own deterministic validation entrypoint,
reporting the result, resuming once, and cleaning up without a product diff,
push, pull request, deployment, or staging mutation.

Acceptance requires claim, resume, report, stop, and cleanup evidence with the
default branch, repository contents, pull requests, Actions configuration,
rulesets, and staging unchanged.

## Slice 5: Bounded E03/E04 Task

Open this slice only after Slice 4 passes. First create a stable repository
Issue that links the existing E03/E04 acceptance documents and case identity,
without the dispatch label; read it back and confirm Symphony has not claimed
it. Adding the dispatch label is a separate, explicit start action.

Keep the issue-only tracker and status identity proven in Slice 2 separate from
both the minimum code-push and Draft pull-request identity and any staging
identity. Never widen or reuse the tracker token for repository content or
pull-request writes. Staging credentials are absent by default, may be injected
only for an explicitly authorized validation step, and must be revoked or
removed independently. Declining staging access must not widen or silently
reuse either GitHub identity.

Use the existing `bio_plot_platform` acceptance authority rather than copying
product instructions into calibration. Symphony may implement, validate,
repair CI and review feedback, and report merge readiness; final merge remains
a human decision.

One successful repository canary is evidence for that repository and the
tested host topology. It does not prove that every repository needs Symphony
or that a cross-repository flywheel is complete.

## Validation and Evidence

Record exact commits, tool versions, commands, exit status, test counts,
service properties, listener addresses, API readback, issue identity,
workspace lifecycle, external writes, human interventions, and elapsed time.
Do not treat static validation, a working dashboard, or a product result alone
as proof of continuation-loop effectiveness.

Keep temporary build and runtime artifacts outside git. Put repository-owned
contracts in their repositories and control-plane secrets outside repository
history. Close this plan with a focused decision that identifies the first
failed gate or the bounded capability actually proven.

## Delivery Order

1. Run Slice 1 as an ephemeral probe and record any newly discovered blocker.
2. Merge this plan after calibration validation and independent review.
3. Run the foreground scratch E2E before creating the bounded host service.
4. Install and verify the bounded service without enabling recurrence.
5. Run Slice 4 through a separate `bio_plot_platform` change only if the
   generic gates pass.
6. Consider Slice 5 only after no-product continuation succeeds.
7. Publish a docs-only closeout; wider orchestration and auto-merge remain
   closed unless later evidence supports them.
