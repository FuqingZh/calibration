# v1.7 AO Repository Adoption Contract Implementation Plan

Status: closed; continuation and stacked delivery validated.

## Purpose

Prevent an AO project record or passing static diagnostic from being mistaken
for an operational repository feedback loop. Make repository adoption
repeatable across opted-in repositories without silently broadening host,
identity, permission, issue-intake, or auto-merge authority.

## Scope

- Add one parameterized, idempotent plan/apply command for an explicitly
  selected repository.
- Require explicit repository path, project name, Codex home, and permission.
- Reject a mismatched project path or an absent, symlinked, or overly
  permissive isolated Codex home.
- Verify persistent service state, daemon readiness, doctor output, persisted
  project configuration, and configuration readback.
- Define `registered`, `configured`, `runtime-ready`, and
  `continuation-proven` as distinct evidence states.
- Route AO onboarding and repair from the calibration skill to the operational
  runbook.
- Add focused unit and behavioral coverage for the observed false-completion
  gap.

## Non-Goals

- bulk-enroll repositories;
- change a target repository's product code;
- enable issue-tracker intake, automatic work discovery, or auto-merge;
- make `bypass-permissions` a cross-host default;
- replace GitHub CI or Automatic Codex Review;
- claim continuation from static checks.

## Validation

1. Run the repository's complete PDM quality gate.
2. Run the legacy installer and diff checks from `AGENTS.md`.
3. Execute a targeted skill smoke for case 10 and inspect routing, evidence
   states, and refusal to overclaim.
4. On the accepted host, run the initializer in plan mode, then apply it to one
   opted-in real repository.
5. Create or claim the worker before its implementation branch or pull request.
6. Complete one anchored Automatic Codex Review correction loop before marking
   that repository `continuation-proven`.
