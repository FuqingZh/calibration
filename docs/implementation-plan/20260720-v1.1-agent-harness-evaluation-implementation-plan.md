# Repository Harness Diagnosis And Evaluation Implementation Plan

Version: v1.1

Date: 2026-07-20

Status: active

## Goal

Fill two cross-project judgment gaps in calibration:

1. diagnose which repository capability is missing when agents repeatedly
   stall or rediscover work, and place that capability with the correct owner;
2. determine whether a skill, harness, or workflow is genuinely better on
   representative work rather than merely valid or more detailed.

Calibration owns the diagnosis and evaluation method. Each project continues
to own its concrete scripts, tests, CI, observability, deployment checks, and
local instructions.

The ownership decision is recorded in
`../decisions/2026-07-20-agent-harness-and-evaluation-ownership.md`.

## Source Direction

The plan distills rather than reproduces OpenAI's published practices:

- Harness Engineering: repository legibility, mechanical feedback,
  observability, and entropy control;
- Symphony: adoption conditions for isolated task workspaces, durable state,
  retries, and orchestration;
- self-improving tax agents: reviewed production evidence progressing through
  findings, evaluations, fixes, and regression checks;
- Codex-maxxing: boundaries between durable task context, memory, repository
  facts, and verifiable goals;
- AI-native engineering teams: delegate execution, review evidence, retain
  ownership, and escalate only decisions that require human authority.

  `AGENTS.md` as part of this plan.

  `WORKFLOW.md`.

- Do not run paid model evaluation on ordinary documentation or validation

### Slice 3: Independent Comparative Evaluation

Start only after Slice 2 merges.

Use the accepted pre-Slice-2 state as baseline and the Slice 2 merge as
candidate. Run both arms against the same expanded case set; do not compare the
new aggregate directly with the earlier 13-case result.

Reuse the isolated A/B harness unless repeated operation demonstrates that a
repository-owned runner is necessary. Freeze both sources, model, reasoning
effort, fixtures, turns, and repetitions. Use fresh contexts and isolated
Codex homes. Complete the blind scorecard before revealing the arm map.

Acceptance gate:

- targeted harness/evaluation cases and the existing regression cases run for
  both arms;
- every run records its source, model, effort, case, repetition, and turn
  identity;
- correctness, required behavior, and critical failures pass before resource
  measures are considered;
- an independent fresh task judges only blind outputs and rubric;
- incomplete score fields fail closed;
- the closeout distinguishes regression acceptance from comparative
  superiority and records limitations.

### Slice 4: Bounded Project Pilots

Start only if Slice 3 supports retaining the candidate.

Pilot the judgment references on real tasks in `workflows` and
`trait_association`:

- use `workflows` to test repeated delivery operations, scripts, dry-run, and
  external readback placement;
- use `trait_association` to test proportional harness decisions and
  repository-map boundaries without importing application-scale machinery.

Make project changes only when task evidence demonstrates a missing
capability. Do not generate `AGENTS.md` files by directory or impose one CI
shape on unrelated repositories.

Acceptance gate:

- each pilot records the observed gap, selected owner, smallest change,
  verification evidence, and remaining project-specific boundary;
- the agent resolves discoverable facts and escalates only material judgment;
- repeated cross-project findings are reviewed before promotion to
  calibration;
- orchestration remains deferred unless its full adoption gate is satisfied.

## Dependency Order

```text
static structural gate (complete)
-> harness/evaluation ownership and references
-> independent behavior evaluation
-> bounded real-project pilots
-> evidence-based promotion only when repeated
```

Each slice is independently reviewable. Do not combine reference authoring,
blind judging, and project pilot changes in one pull request or task context.

## Risks And Controls

- **Prompt accretion:** keep detailed judgment in on-demand references and add
  only routing text to the skill.
- **False improvement claim:** separate validation, evaluation machinery, and
  comparative behavior evidence.
- **Case-set drift:** preserve existing cases, version the expanded set, and
  run both arms on the same definitions.
- **Self-confirming evaluation:** use a fresh blind-judging task without source
  identity, intended change, or previous conclusion.
- **Public evidence leakage:** use synthetic or sanitized fixtures and
  reconstructable references.
- **Overbuilt harnesses:** scale capabilities to demonstrated project failures
  and keep orchestration behind its adoption gate.
