# Repository Delivery Feedback Loop Implementation Plan

Version: v1.3

Date: 2026-07-21

Status: active candidate

## Goal

Add the smallest cross-project rule needed to turn repository setup, CI,
agent review, and cloud execution failures into durable feedback-loop
improvements without preconfiguring every repository.

## Boundaries

- Keep the existing calibration skill and repository harness as owners.
- Do not add a skill, universal `AGENTS.md` template, delivery schema,
  orchestrator, or per-repository scheduled task.
- Keep the existing behavioral cases unchanged during smoke validation.
- Prefer repository-owned setup and validation entrypoints over duplicated
  cloud configuration.

## Slice 1: Candidate Guidance And Local Adoption

- Route repository delivery feedback to `discipline/harness.md`.
- Add the default-first, failure-driven delivery feedback loop.
- Record external control-plane ownership and authorization boundaries.
- Add the smallest useful `calibration` repository `AGENTS.md` increment.
- Expose validation dependencies through standard `requirements.txt` and make
  CI consume the same entrypoint.

### Local acceptance

- unit tests and `scripts/validate_skills.py` pass;
- JSON behavior inputs remain unchanged;
- `bash -n install.sh` and isolated installer dry-run pass;
- local reference links resolve;
- `git diff --check` passes;
- `error.log` and other unrelated user files remain untouched.

## Slice 2: Real Cloud Smoke

1. Run a read-only task in `FuqingZh/calibration` against this candidate after
   the candidate branch is available to Codex cloud.
2. Run the same evidence-oriented smoke protocol in one other existing cloud
   environment.
3. Require both tasks to leave no diff and classify setup, repository, or
   external-control-plane gaps accurately.

If the second repository does not support the same cross-project direction,
keep the finding project-local and do not accept the calibration behavior.
If both smokes support it, close this plan with the task IDs, environment
labels, observed commands, and remaining limitations.

## Delivery

Keep the pull request Draft until both smoke results are reviewed. Static
validation proves the candidate is internally consistent; it does not by
itself prove the delivery feedback loop generalized.
