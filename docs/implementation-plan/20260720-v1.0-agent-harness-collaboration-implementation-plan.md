# Agent Harness And PR Collaboration Implementation Plan

Version: v1.0

Date: 2026-07-20

Status: ready after pull request #3 merges

## Goal

Turn the accepted skill guidance into a repository-owned, PR-centered feedback
loop: practical agent instructions, deterministic validation, maintainable
behavior evaluations, and independent review evidence.

The plan applies the repository-legibility and enforceable-feedback direction
from [Harness Engineering](https://openai.com/index/harness-engineering/), the
practical `AGENTS.md` contract from
[Codex best practices](https://learn.chatgpt.com/guides/best-practices), and the
progressive-disclosure and invocation guidance from
[Build skills](https://learn.chatgpt.com/docs/build-skills).

## Non-Goals

- Do not change skill bodies in the validation and collaboration PRs.
- Do not add a new skill for writing `AGENTS.md`.
- Do not run paid model evaluation on every ordinary documentation PR.
- Do not commit raw model transcripts, isolated Codex homes, private arm maps,
  credentials, or user configuration.
- Do not migrate repositories to an organization or create a Codex machine
  user as part of this plan.

## Execution Slices

### Slice 1: Static Validation Gate

Branch: `codex/ci/calibration-skill-validation`

Own these surfaces:

- `scripts/validate_skills.py`
- `tests/test_validate_skills.py`
- `.github/workflows/validate-skills.yml`
- minimal dependency metadata only if the validator requires it

The validation gate must discover the first-party and vendored skills from the
repository rather than maintain an unrelated duplicate list. The Python
validator must check:

- portable `SKILL.md` frontmatter and required fields;
- valid `agents/openai.yaml` structure, prompt references, description lengths,
  and the intended implicit-invocation policy;
- parseable behavioral `test-prompts.json` files;
- absence of retired `disable-model-invocation` fields;
- referenced repository-local files that are required by an active skill.

The workflow must also run `bash -n install.sh`,
`bash install.sh --dry-run`, and `git diff --check` against the pull request
base or pushed commit range.

Acceptance gate:

- focused validator tests pass locally and in GitHub Actions;
- the workflow runs on pull requests and pushes to `main`;
- after the first successful run, the existing default-branch ruleset requires
  the validation check;
- a deliberately malformed fixture proves each important failure path.

### Slice 2: Repository Agent And PR Contract

Branch: `codex/docs/calibration-agent-pr-contract`

Own these surfaces:

- root `AGENTS.md` with repository layout, supported validation commands, PR
  expectations, and completion criteria;
- an on-demand reusable reference under
  `references/engineering/docs/workflow/` for writing repository `AGENTS.md`;
- `references/engineering/docs_index.md` routing to that reference;
- `.github/pull_request_template.md`;
- a compact review reference when rules beyond the PR template are justified.

Keep the root `AGENTS.md` concrete and short. It must not duplicate the full
`$calibration` principles or generic OpenAI documentation. The PR template must
ask for scope, contract impact, validation evidence, evaluation impact,
limitations, and `Agent involvement`.

Acceptance gate:

- a fresh Codex task reports the expected global and repository instruction
  chain;
- every documented command exists and succeeds;
- the PR template renders correctly on a test PR;
- no new skill or duplicated engineering rule is introduced.

### Slice 3: Maintainable Skill Evaluation Harness

Branch: `codex/test/calibration-skill-eval-harness`

Promote deterministic assets into `evals/skills/`:

- runner and case-schema validation;
- synthetic fixtures and behavior rubrics;
- blind-pair generation and scorecard validation;
- runner unit tests and a network-free dry-run path.

Keep generated results outside Git. The harness must require an explicit model
execution flag and must refuse to summarize an incomplete scorecard.

Repair the current cases:

- stop `brainstorming` at the first Design Checkpoint, with a later turn used
  only when checkpoint or approval-wait behavior remains unresolved;
- add multi-turn `grilling` cases for confirmation, dependent decisions, no
  implementation before confirmation, and the stopping summary;
- add `writing-code-docs` cases that distinguish several short examples from
  one overloaded combination and that assess caller-facing example length;
- keep holdout cases separate from cases used to author a skill change.

Acceptance gate:

- existing runner tests and the new completeness tests pass;
- a dry run produces the expected matrix without contacting a model;
- interrupted and resumed runs preserve case, arm, model, effort, and turn
  integrity;
- incomplete scoring fails closed with an actionable message.

### Slice 4: Measured Skill Optimization

Start only after Slices 1-3 merge.

For each hypothesis, create one focused PR and record:

```text
official guidance -> current owner -> measured gap -> smallest change -> eval case
```

Audit descriptions and trigger boundaries first because they determine skill
discovery under progressive disclosure. Then inspect duplicated instructions
and conditional references. Preserve measured product behavior such as the
`brainstorming` gate even when shorter text is possible.

Use the pull request #3 merge result as the baseline. Compare the same model,
reasoning effort, fixtures, and repetitions in fresh contexts. Complete blind
scoring before revealing the arm map. Resource savings count only when required
behavior and critical-failure gates still pass.

## PR And Dependency Order

1. Static validation gate; produces the required CI check.
2. Agent and PR contract; consumes the validated command surface.
3. Evaluation harness; consumes CI and PR evidence conventions.
4. One PR per measured skill hypothesis; consumes the accepted harness.

Do not combine these slices. Each can be rejected or revised without undoing
the others.

## Risks And Controls

- **Public repository leakage:** synthetic fixtures only; keep raw runs and
  private maps out of Git.
- **Self-confirming evaluation:** separate authoring and blind-judging tasks;
  preserve holdouts.
- **Prompt accretion:** assign each rule one owner and add text only for a
  measured gap.
- **CI without behavioral evidence:** static validation catches structure, not
  output quality; keep model evaluation as a separate explicit gate for skill
  behavior changes.
- **Ruleset lockout:** require the new status check only after it has succeeded
  on a pull request and its stable check name is known.
