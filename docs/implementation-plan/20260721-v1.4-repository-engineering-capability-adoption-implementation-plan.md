# Repository Engineering Capability Adoption Implementation Plan

Version: v1.4

Date: 2026-07-21

Status: closed

## Goal

Extend the accepted repository delivery feedback loop with a proportional
entrypoint for assessing and adopting agent-led engineering capabilities across
different repositories. An agent should discover local authority and the
current feedback loop, leave adequate capabilities alone, and add only the
smallest evidenced gap needed to advance a representative task.

## Non-goals

- Do not add a new skill, universal repository template, maturity score, or
  mandatory artifact checklist.
- Do not require every repository to have CI, a cloud environment, a runtime
  UI, staging, Kubernetes, an orchestrator, or the same `AGENTS.md` structure.
- Do not encode `bio_plot` deployment, permissions, labels, or rollback details
  as cross-project rules.
- Do not build a custom pull-request babysitter or orchestration engine when a
  platform-native capability or Symphony can provide the execution surface.
- Do not enable auto-merge or change the existing calibration behavior prompts
  or frozen historical evaluation fixtures.

## Contract Surfaces

- `references/engineering/discipline/harness.md` owns the evidence-driven
  capability assessment, minimal adoption, and orchestration boundary.
- `skills/calibration/SKILL.md` routes explicit repository assessment and
  adoption work to that owner without widening the skill's trigger.
- Deterministic contract tests protect proportionality and the distinction
  between cross-project judgment and repository-local implementation.
- Real repositories supply behavioral evidence; they do not become templates.

## Slice 1: Calibration Adoption Entry Point

Add the repository capability adoption workflow and the positive side of the
orchestration gate. Close the accepted v1.3 delivery-loop records and make this
plan the current docs entrypoint.

Acceptance:

- repository capabilities are classified from current evidence as present,
  missing, or not applicable;
- adequate and not-applicable capabilities do not create changes;
- a missing capability receives the smallest increment owned by the repository
  or external control plane;
- application runtime observability and orchestration remain conditional;
- the existing behavior prompts and historical evaluation fixtures are
  unchanged; and
- unit tests, skill validation, installer checks, isolated dry-run, reference
  checks, and diff checks pass.

This slice is one calibration pull request and produces the guidance consumed
by both live pilots.

## Slice 2: `bio_plot` Representative Application Pilot

Use the repository's existing E03/E04 acceptance authority as the first real
task. Symphony is the intended canary engine because the user's parallel task
load and resume/repair need establish the initial orchestration trigger; they do
not by themselves satisfy the full adoption gate.

Before starting the canary, read back evidence for the remaining gate:

- E03/E04 have a stable task state and repository-owned acceptance authority;
- the execution path provides an isolated workspace and reliable permission
  boundary; and
- a runnable Symphony path will actually claim, resume, and report this task.

If any condition is unavailable, record a `NO-GO` with its owner and stop the
orchestration attempt. Do not build a custom substitute. On an execution
failure, preserve durable repository and staging evidence and use only the
owning repository or control plane's existing cleanup and rollback contract.

The pilot should let the execution agent discover the repository's local
validation, staging, permissions, pull-request, and rollback contracts. It must
not receive a calibration-authored Kubernetes or label SOP. It advances the
task through implementation, runtime evidence, pull-request validation,
independent review, and feedback repair to merge-ready; final merge remains a
human decision.

Record the issue/task identity, commits, PR, CI and review outcome, runtime
evidence, human interventions, orchestration failures, and limitations. A
working product change without a functioning continuation loop is partial
evidence, not proof that orchestration succeeded.

## Slice 3: `biofetch` Transfer Check

Run the same repository capability assessment in the Go/CLI repository. Judge
only capabilities material to its actual delivery path. Do not require a UI,
staging, Kubernetes, or Symphony merely because the application pilot used
them.

If the assessment finds no evidenced gap, a no-change result is valid. If it
finds a concrete gap, keep the remedy and verification in `biofetch` and land
it through a separate repository pull request.

## Slice 4: Evidence Closeout

Compare the two pilots for transferable judgments: authority discovery,
verification, delivery feedback, application observability when applicable,
orchestration adoption, and human escalation. Promote only behavior supported
across its stated applicability boundary. Keep project mechanisms in their
owning repositories or control planes.

Use a focused closeout rather than a broad blind A/B. If either pilot fails,
record the failing stage and limitation before changing calibration again.

## Delivery Order

1. Merge Slice 1 after deterministic validation and independent review.
2. Validate the full orchestration gate, then run Slice 2 as the first real
   Symphony canary or record a bounded `NO-GO`.
3. Run Slice 3 without copying application-only requirements.
4. Close this plan from the combined evidence; only then consider wider
   orchestration, recurring gardening, or auto-merge.

## Closeout

This plan closed with the decision recorded in
`../decisions/2026-07-21-repository-engineering-capability-adoption-closeout.md`.

Slice 1 merged through calibration pull request #11. Slice 2 stopped at its
predeclared orchestration gate: E03/E04 had repository acceptance authority,
but no claimable tracker task, repository workflow, runnable Symphony engine,
isolated engine-owned workspace, or least-privilege execution identity was
available. No canary ran and no substitute was built.

Slice 3 classified application-only capabilities as not applicable to
`biofetch`, retained its adequate local contracts, and found one delivery gap:
GitHub had no mechanical execution of the documented Go gate. A separate
repository pull request #4 added the minimal CI workflow, established the
stable `validate-biofetch` check with a successful real run, and then added
that check to the existing default-branch ruleset. It merged as
`eec47579411cdb8aee9b385ef19fc293d02bd9ae`, and the same check passed on
`main`.

The combined evidence supports retaining the proportional adoption guidance.
It does not establish Symphony effectiveness; the closeout's explicit reopen
conditions must be satisfied before another canary attempt.
