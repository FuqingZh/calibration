# Step-to-Step Artifact Flow Specification v1.1

- Date: `2026-03-04`
- Scope: 通用 stage 串联与 artifact 交接规范

## 1. 总体目标

1. 明确每个 stage 对外暴露哪些 artifact。
2. 明确下游如何稳定读取上游产物。
3. 避免下游对上游目录细节和临时文件产生耦合。

## 2. Public / Private Artifact 边界

Public（可被下游消费）：

1. `out/` 中被登记到 `IO.Out.Files` 的文件
2. `meta/meta.json`

Private（禁止下游依赖）：

1. `tmp/**`
2. `logs/**`
3. `out/` 内未登记到 `IO.Out.Files` 的临时产物

## 3. 统一交接协议

下游不得猜测上游目录细节。推荐做法是：

1. 调度层或 orchestrator 先读取上游 `meta/meta.json`（stage manifest）。
2. 解析 `IO.Out.Root` 与 `IO.Out.Files`。
3. 将显式 artifact 路径作为 `--file_in_*` / `--dir_in_*` 传给下游 stage。
4. 仅在确有必要时，才把上游 `meta/meta.json` 本身继续向下传递。

下游通过 manifest 解析 artifact 的标准步骤：

1. 读取上游 `meta/meta.json`
2. 解析 `IO.Out.Root`
3. 通过 `IO.Out.Files` 定位需要的 artifact
4. 结合 `Config.Effective` 继承必要上下文

## 4. 推荐标准链路（示例）

下述链路是“模板”，可替换 stage 名称但不改交接协议：

`intake -> ident -> quant -> enrichment -> network_layer -> plot_renderer -> report`

并行分支可表示为：

`quant -> (enrichment, network_layer) -> plot_renderer`

## 5. 各 stage 的最小交接清单（示例）

### 5.1 intake

上游输入：原始输入文件 + TOML 配置  
对外输出键：

1. `DataPrimary`
2. `DataSecondary`
3. `InfoManifest`

### 5.2 ident

上游输入：`intake` 的数据主表  
对外输出键：

1. `DataLinked`
2. `InfoIdentifierMap`
3. `InfoQualityReport`

### 5.3 quant

上游输入：`ident` 的关联主表  
对外输出键：

1. `DataQuantSummary`
2. `DataQuadrantTable`
3. `DataFeatureSets`

### 5.4 enrichment

上游输入：`quant` 的特征集  
对外输出键：

1. `DataEnrichmentMain`
2. `DataEnrichmentByGroup`

### 5.5 network_layer

上游输入：`quant` 主表和映射信息  
对外输出键：

1. `DataNetworkNodes`
2. `DataNetworkEdges`

### 5.6 plot_renderer

上游输入：`quant` + `enrichment` + `network_layer`  
对外输出键：

1. `GraphManifest`
2. `GraphMainSet`

### 5.7 report

上游输入：`plot_renderer` 图形集合 + 各类摘要表  
对外输出键：

1. `ReportMain`
2. `ReportAssetsDir`

## 6. 文件命名约束

### 6.1 Parquet 落地决策规则

1. 单逻辑表使用 `data-*.parquet` 单文件，并在 `IO.Out.Files` 中登记为文件 artifact。
2. 分区化主数据使用 `data-*/` Parquet dataset，并在 `IO.Out.Files` 中登记为目录 artifact。
3. 只有在存在明确分区键且下游会利用分区裁剪时，才允许使用 dataset。
4. 使用 dataset 时，必须在文档和 `meta/meta.json`（stage manifest）中显式声明 `PartitionBy`。
5. 下游不得把 `data-*.parquet` 与 `data-*/` 混为同一种 artifact 处理。

### 6.2 通用命名

1. 数据：`data-*.parquet|tsv`
2. 信息：`info-*.tsv|json`
3. 绘图输入：`plot-*.tsv|parquet`
4. 图形：`graph-*.png|pdf|svg`
5. 打包：`bundle-*.zip|tar.zst`（可选）
6. 前端：`frontend-*.*`（可选）
6. `-` 表示语义段分隔，`_` 表示词组连接

附加约束：

1. `plot-*` 默认用于绘图或报告层输入，不作为主分析真源。
2. `plot-*` 默认保存为 `TSV`；仅当 `TSV` 的体积或 IO 成本不可接受时，才升级为 `plot-*.parquet`。
3. `bundle-*` 与 `frontend-*` 仅用于交付或展示，不作为计算真源。
4. 下游计算依赖应回到 `data-/info-/plot-/graph-` 体系。
5. 多组图默认按目录分组输出：`out/<GroupSetId>/graph-*.{fmt}`；若同时按组与通路出图，则使用 `out/<GroupSetId>/<PathwayId>/graph-*.{fmt}`。

## 7. 反模式（禁止）

1. 下游直接消费上游 `tmp/` 内容。
2. 用固定相对路径假设代替 `meta` 交接。
3. 在 `meta.json` 存储会改变计算逻辑的参数。
4. 各 stage 自定义不一致的文件命名前缀。
5. 把 `frontend-*` 当作不可追溯且不可重建的唯一输入。
