# Identifier Specification v1.1

- Date: `2026-03-04`
- Scope: 通用实体技术主键规范（去领域绑定）

## 1. 目标

为任意结构化实体定义统一技术主键，要求：

1. 唯一
2. 可逆
3. 跨语言可实现
4. 可稳定用于 join / merge / 去重

## 2. 核心字段

1. `_EntityId`：技术主键（系统内部主键）
2. `EntityKind`：实体类型（如 `sample`、`feature`、`edge`）
3. `NaturalKey*`：业务字段（可多个）

说明：

1. `_EntityId` 用于技术 join。
2. 业务展示与业务语义由 `NaturalKey*` 承担。

## 3. 编码协议

### 3.1 Canonical Payload

输入结构：

```json
{
  "kind": "<EntityKind>",
  "keys": {
    "<field_a>": "<value_a>",
    "<field_b>": "<value_b>"
  }
}
```

约束：

1. `keys` 的键按字典序排序后再序列化。
2. 使用 UTF-8 编码。
3. JSON 使用紧凑格式（无多余空格）。

### 3.2 `_EntityId` 生成

1. `payload_json = canonical_json(payload)`
2. `payload_b64 = base64(payload_json)`
3. `_EntityId = "B64J:" + payload_b64`

## 4. 解码协议

1. 验证前缀必须为 `B64J:`
2. 去前缀后做 base64 解码
3. JSON 反序列化得到 `kind` 与 `keys`

## 5. 为什么采用该方案

1. 不依赖分隔符猜测，避免字符串拼接歧义。
2. 可逆，不损失原始业务键。
3. 结构稳定，适合 Parquet / TSV / JSON 传递。
4. 跨语言实现简单（Python/R/Rust/Go 都可直接实现）。

## 6. 反模式（禁止）

1. 用 `field1_field2` 直接拼接作为主键。
2. 把领域语义（如平台、物种、算法模式）塞进 `_EntityId`。
3. 使用不可逆 hash 作为唯一主键且不保留可逆信息。

## 7. 命名关系建议

1. 跨模块统一技术主键列：`_EntityId`
2. 领域模块可定义别名（例如 `_ProteinId`），但必须可映射到 `_EntityId`
3. 下游跨模块 join 优先 `_EntityId`
