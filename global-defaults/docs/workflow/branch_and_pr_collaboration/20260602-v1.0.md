# Branch and PR Collaboration

Version: v1.0
Date: 2026-06-02
Status: current

This document defines the default cross-project branch workflow. Repository-local
rules may be stricter, especially for release branches, regulated deployment
paths, or multi-version maintenance.

## Default Model

Use trunk-based development by default:

```text
protected main + short-lived work branches + PR gate + optional staging/test verification
```

Do not introduce a long-lived `develop` branch by default. Use `develop`,
`release/*`, or release trains only when a repository has a documented need for
multi-version maintenance, fixed release windows, or a shared long-running
integration branch.

## Branch Roles

- `main`: protected trunk. Keep it deployable, testable, and bisectable.
- `codex/*`: short-lived Codex or agent work branches.
- `<type>/*`: short-lived human work branches.
- `staging` or `test`: optional integration or deployment targets.
- `release/*`: optional version maintenance branches when repository policy
  requires them.
- `hotfix/*`: expedited production fix branches when repository policy allows a
  hotfix path.

## Where Work Happens

Create work branches from the latest `main`.

```text
main -> work branch -> local checks -> PR -> verification -> main
```

Develop and commit on the work branch. Do not develop directly on `main`,
`staging`, `test`, `production`, or `release/*` unless a repository-local release
workflow explicitly allows it.

Keep work branches short-lived and focused on one reviewable change. If a branch
drifts materially from `main`, update it before final review.

## Branch Naming

Codex-created branches:

```text
codex/<type>/<scope>-<short-topic>
```

Human-created branches:

```text
<type>/<scope>-<short-topic>
```

The branch name must match the work:

- `type` sets the expected verification focus.
- `scope` identifies the affected repo, product, module, or subsystem.
- `short-topic` identifies the concrete review target.

See `references/naming.md` for the shared branch type list and naming
boundaries.

## Type and Verification

The branch `type` sets the expected verification focus.

| Type | Verification focus |
| --- | --- |
| `feat/*` | verify the new behavior, workflow, API, CLI option, or user-visible capability |
| `fix/*` | verify the failing behavior is corrected and relevant regressions are covered |
| `docs/*` | verify documentation consistency; runtime tests are not required unless docs generation is affected |
| `refactor/*` | run tests proving public behavior is preserved |
| `test/*` | verify the intended test behavior or coverage change |
| `chore/*` | run checks relevant to the touched files |
| `ci/*` | verify workflow, build, deployment, or repository automation behavior |
| `perf/*` | record the performance signal, benchmark context, or scale used for validation |
| `revert/*` | verify the revert restores the intended previous behavior |
| `hotfix/*` | use the smallest safe change and record the reason for expedited handling |

## Testing and Staging

Test on the work branch by default. Run local tests, lint, builds, smoke checks,
or focused validation before merge.

When a shared environment is needed, deploy the work branch or candidate PR to a
`staging` or `test` environment.

`staging` and `test` branches are optional and should be used only as integration
or deployment targets:

- Do not develop directly on `staging` or `test`.
- Do not let `staging` or `test` accumulate unrelated development.
- Do not merge `staging` into `main` by default.
- Prefer merging the original reviewed work branch into `main` after validation.

## Commits

Commit focused changes to the work branch. Commit messages may follow the PR
title style:

```text
<type>(<scope>): <summary>
```

Examples:

```text
fix(bio-plot): reduce chat fallback latency
docs(engineering-canon): define branch policy
feat(bioextract): add KEGG resource builder
```

Small single-purpose PRs may use one commit and squash merge. Larger PRs may
retain multiple commits only when each commit has independent review value.

## Main Protection

`main` is protected. Direct push to `main` is not allowed by default.

All ordinary changes should enter `main` through a PR. Emergency bypasses require
explicit approval and must be followed by a PR or written trace explaining what
was changed, why the bypass was needed, and how it was verified.

## Merge Policy

Merge reviewed and verified work branches into `main`.

Allowed merge actors:

- repository maintainers using the hosting platform UI
- repository maintainers using approved CLI tooling
- platform-native auto-merge or merge queue
- CI/CD or bot automation that obeys branch protection

All merge actors must satisfy the same gates:

- required checks pass
- required reviews or owner approval are satisfied
- stale PR branches are updated or validated by merge queue
- deployment-sensitive verification is recorded

Prefer platform-native auto-merge, merge queue, or merge trains over custom
scripts that push to `main`.

Avoid:

- CI jobs that directly push to `main`
- bots that bypass branch protection
- local merges followed by direct push to `main`
- automatic `staging` to `main` merges that hide which reviewed branch is being
  accepted

After merge, delete short-lived work branches unless repository policy requires
keeping them.

## Summary

Develop and commit on a short-lived work branch from `main`. Test on that branch
or deploy it to a staging/test environment. Merge through a PR gate into
protected `main`. Human and automated merge paths are both acceptable when they
obey the same protection, review, and verification rules.
