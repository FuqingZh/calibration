# Project Docs Architecture

Version: v1.0
Date: 2026-05-27
Status: current

This document defines the default cross-project structure for durable project
documentation. Repository-local documentation may be stricter, but new project
docs should follow this shape unless there is a clear local reason not to.

## Purpose

Project `docs/` stores long-lived knowledge:

- architecture and domain boundaries
- public contracts and artifact schemas
- implementation and migration plans
- testing, smoke, benchmark, and validation standards
- stable decisions that future contributors or agents should reuse

Do not use `docs/` for raw execution transcripts, temporary command output, or
unprocessed task logs. Task-level retrospectives belong in `.traces/`.

## Recommended Layout

Use the smallest structure that fits the project. Do not create empty
directories just to match this list.

```text
docs/
  README.md
  architecture/
  implementation-plan/
  testing/
  benchmarks/
  assets/
  archive/
```

Directory roles:

- `docs/README.md`: entrypoint, reading order, current authority map.
- `architecture/`: durable architecture, domain boundaries, contracts, data
  flow, artifact layout, and stable design decisions.
- `implementation-plan/`: PR slices, phased plans, migration plans, and
  closeout notes.
- `testing/` or `development/testing.md`: test strategy, smoke commands, real
  data validation, and acceptance criteria.
- `benchmarks/`: performance, numerical consistency, real-data baselines, and
  reproducibility context.
- `assets/`: images and diagrams referenced by docs.
- `archive/`: superseded or historical material with clear traceability value.

## README Requirements

`docs/README.md` should tell a new contributor or agent:

- what to read first
- which architecture document is current
- where the active implementation plan lives
- where testing and validation standards live
- which documents are historical or superseded

The README is navigation, not a replacement for the source documents.

## Relationship To `AGENTS.md`

The canonical `AGENTS.md` content boundary is defined by the
[repository harness](../../../discipline/harness.md). Project `docs/` remain
the durable authority, with `docs/README.md` as their navigation entrypoint.

## File Naming

For new dated documents, use lowercase kebab-case:

```text
YYYYMMDD-vMAJOR.MINOR-topic-kind.md
```

Examples:

```text
20260527-v1.0-project-docs-architecture.md
20260527-v1.0-task-traces-and-retros.md
20260527-v1.1-gwas-report-template-alignment-plan.md
```

`topic` names the domain object. `kind` names the document type.

Recommended `kind` values:

- `architecture`
- `contract`
- `pr-roadmap`
- `test-plan`
- `benchmark`
- `implementation-plan`
- `migration-plan`
- `decision-record`
- `closeout`

Avoid:

- spaces, colons, and Chinese punctuation in file names
- mixed case in long-lived document names
- temporary names such as `tmp.md`
- undated, unversioned long-lived documents, except `README.md`

Existing documents do not need a bulk rename. Apply this rule to new documents
and migrate old names only during natural edits.

## Document Lifecycle

At the top of durable documents, prefer a small status block:

```text
Version: v1.0
Date: 2026-05-27
Status: current
```

When a document is replaced, mark it clearly:

```text
Status: superseded by <relative/path/to/new-doc.md>
```

Stable contracts should move into `architecture/`, `testing/`, or
`benchmarks/`. One-off exploration should stay out of `docs/` unless it has
become a benchmark, test plan, decision record, or reusable implementation
plan.

## Relationship To `.traces`

Use `.traces/` for task-level execution retrospectives and short-lived
observations. Promote a trace observation into `docs/` only after it has become
a stable project rule, contract, test standard, or implementation decision.
