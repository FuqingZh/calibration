# Main Path Readability v1.0

- Date: `2026-03-18`
- Scope: 通用工程实现中的业务主路径可读性原则

## 1. 核心结论

业务主路径应当保持短、直、可线性复述。

推荐经验阈值：

1. 顶层应当直接暴露阶段顺序。
2. 下钻 1 层应当看到每个阶段的真实职责。
3. 下钻 2 层应当接近实际动作。
4. 若理解主逻辑通常需要第 3 次连续下钻，设计大概率已经失去直接性。

该阈值针对“理解主流程所需的连续下钻深度”，而不是机械统计函数调用层数。

## 2. 审计对象

应审计的是：

1. 读者为了理解主业务流程，必须连续跳转多少层。
2. 每次跳转是否带来新的语义、边界、约束或复用价值。
3. 顶层是否仍可线性复述，而不是被泛动词壳函数掩盖。

不应仅按调用栈层数判定优劣。

## 3. 主路径可读性规则

业务主路径建议满足：

1. 0 次跳转即可看懂阶段顺序。
2. 1 次跳转即可看懂该阶段做什么。
3. 2 次跳转即可接触实义逻辑。
4. 不应把核心分支、核心筛选或核心调度藏在连续三层转发之后。

不推荐的主路径形态：

1. `main -> run -> execute -> process -> handle`
2. 每层都只做一层转发而不新增语义。
3. 用泛动词隐藏实际步骤与边界。

## 4. 空转抽象

真正应反对的不是“层数本身”，而是“空转抽象”。

空转抽象的典型特征：

1. 每跳一层都要求读者付出跳转成本。
2. 但该层没有新增语义、边界、约束或复用价值。
3. 逻辑真实入口被推迟到多层之后。

判断一层抽象是否合理的标准：

1. 是否新增了独立语义。
2. 是否形成了清晰边界。
3. 是否带来了稳定复用价值。
4. 是否显著降低了顶层噪音，而不是仅改变了函数名。

若以上均不成立，应优先内联该层。

## 5. 合理例外

以下情形允许更深的调用层次，但不应破坏主路径直达性：

1. CLI 入口、参数解析、依赖注入、日志包装等基础设施层。
2. orchestration / domain / storage 等明确分层，且每层都有真实信息增量。
3. 递归、遍历、解释器、编译器等天然层次较深的问题结构。

即便存在例外，业务骨架也应尽量在顶层可复述。

## 6. 推荐重构动作

遇到主路径下钻过深时，优先考虑：

1. 删除纯转发层。
2. 把顶层函数收成可直接复述的阶段骨架。
3. 将第二层函数改为实义函数，而不是泛动词壳函数。
4. 将关键分支与调度逻辑提升到更可见的位置。

推荐的第二层命名风格：

1. `load_sample_table`
2. `normalize_intensity_matrix`
3. `fit_limma_model`
4. `write_enrichment_report`

避免：

1. `run`
2. `execute`
3. `process`
4. `handle`

## 7. 一句话规则

如果业务主逻辑经常需要连续三次下钻才能看见，作者通常不是在抽象，而是在藏逻辑。
