# Pipeline Directory Layout Specification v1.1

- Date: `2026-03-04`
- Scope: 通用 step-based pipeline 目录规范
- Alignment: `axiomkit.workspace.WorkspacePlan`

## 1. 设计原则

1. 一个 stage 对应一个独立目录根（self-contained）。
2. 目录结构可按项目实际阶段形态调整；稳定目录约定用于减少协作与自动化歧义。
3. 下游交接方式由项目契约决定；`meta/meta.json` 可作为推荐的机器可读索引之一。
4. 产物、日志、临时文件隔离，确保可审计与可清理。

## 2. 推荐目录结构（按项目调整）

每个 stage 根目录可根据实际情况调整。若项目没有更具体约定，建议包含：

1. `in/`
2. `out/`
3. `tmp/`
4. `logs/`
5. `configs/`
6. `meta/`

建议形态：

```text
<dir_workspace>/<module_or_stage>/
├── in/
├── out/
├── tmp/
├── logs/
├── configs/
└── meta/
    └── meta.json
```

## 3. 子目录语义

1. `in/`：输入快照或输入链接（可选写入，不能当下游真源）。
2. `out/`：对外可交付产物（仅稳定结果）。
3. `tmp/`：临时中间文件（可随时清理，不可下游依赖）。
4. `logs/`：运行日志。
5. `configs/`：该 stage 的 TOML 配置与配置快照。
6. `meta/`：可选的机器可读索引入口（`meta.json`，即 stage manifest），是否作为唯一交接入口由项目契约决定。

## 4. out/ 内部约定

推荐按用途分层：

1. `out/tables/`
2. `out/figures/`
3. `out/artifacts/`

输出文件命名必须遵循前缀规范：

1. `data-...`
2. `info-...`
3. `plot-...`
4. `graph-...`
5. `bundle-...`（可选，归档打包）
6. `frontend-...`（可选，前端直连）

Parquet 落地约束：

1. 单逻辑表优先写为 `out/data-*.parquet`。
2. 仅当产物是分区化主数据时，才写为 `out/data-*/` dataset 目录。
3. dataset 目录建议在项目约定的 manifest 或结果索引中按目录登记，并额外声明分区键。
4. 多组图默认按组子目录输出：`out/<GroupSetId>/graph-*.{fmt}`。
5. 若图形同时按组和通路拆分，则使用 `out/<GroupSetId>/<PathwayId>/graph-*.{fmt}`。

语义约束：

1. `data-/info-/plot-/graph-` 是核心产物前缀。
2. `data-*` / `Parquet` 是稳定 typed table 的主计算真源。
3. `TSV` 优先用于人工阅读、快速人工校验、绘图输入和报告输入；仅当文本可读性比类型保真更重要时，才优先选择 `TSV`。
4. `plot-*` 默认保存为 `TSV`，仅当 `TSV` 的体积或 IO 成本不可接受时，才升级为 `plot-*.parquet`。
5. `TSV` 不应替代 `Parquet` 成为大体量、类型敏感或契约关键表的主计算来源。
6. `bundle-/frontend-` 是保留扩展前缀，不作为下游计算真源。

## 5. meta/meta.json 建议契约

`meta/meta.json` 在语义上是 stage manifest；当前路径命名保留为 `meta/meta.json`。它承担索引和交接契约职责，不承担语义配置真源职责。


若项目选择以 `meta/meta.json` 作为 stage manifest，建议包含：

1. `Self`：`Name`、`Version`、`SchemaVersion`、`Status`、时间戳
2. `IO.In`：输入来源（路径或上游 meta）
3. `IO.Out.Root`：当前 stage 根目录绝对路径
4. `IO.Out.Files`：跨模块需要的关键文件相对路径
5. `Config.Effective`：最终生效配置

## 6. 配置文件规范

1. 配置统一使用 TOML。
2. 配置建议放置于 `configs/stage.toml` 或 `configs/pipeline.toml`。
3. 禁止使用 YAML 作为语义配置真源。

## 7. 反模式（禁止）

1. 在没有项目约定的情况下，把机器交接 manifest 混放到不稳定结果目录中。
2. 把临时缓存文件写进 `out/`。
3. 让下游直接读取上游 `tmp/`。
4. 在脚本中硬编码上游输出相对路径，且没有任何显式配置、manifest 或调度层 artifact 绑定。
5. 让下游计算直接依赖 `bundle-*` 或 `frontend-*`。
